9주차
MVC 흐름 · ViewResolver · Model
@Controller · @GetMapping · return · Model.addAttribute
패턴 ①★★★
@GetMapping("/")
public String home(Model model) {
    model.addAttribute("msg", "안녕!");
    return "home";
}
ViewResolver → /WEB-INF/views/home.jsp 실행
JSP의 ${msg} → "안녕!" 출력
return "home" = View 이름. prefix + "home" + suffix 로 실제 경로 조합.
application.properties: prefix=/WEB-INF/views/ suffix=.jsp
패턴 ② return 변형★★★
// 케이스 A: return "home"
return "home";
// → ViewResolver 거쳐 home.jsp 렌더링
// → URL은 그대로 유지됨

// 케이스 B: return "redirect:/"
return "redirect:/";
// → ViewResolver 안 거침
// → 브라우저가 "/" 로 새 GET 요청
// → URL이 "/" 로 바뀜
A: URL 유지, JSP 렌더링 | B: URL "/" 로 변경, 새 요청
MVC 전체 흐름★★★
클라이언트→ DispatcherServlet→ HandlerMapping→ HandlerAdapter→ Controller→ Service→DAO→DB
결과→ Controller (return "뷰명")→ ViewResolver→ View (JSP)→ 브라우저 응답
HandlerMapping
@GetMapping 애너테이션을 읽어 어떤 Controller 메소드를 실행할지 결정
HandlerAdapter
실제로 Controller의 메소드를 호출하는 실행 담당 객체
ViewResolver
return "home" → /WEB-INF/views/home.jsp 경로로 변환
⚠
Controller의 return은 View 이름만 반환 가능. 데이터는 반드시 Model을 통해 전달. return으로 데이터를 넘기려 하면 안 됨!
10주차
@RequestParam · VO 자동 매핑 · @Autowired
파라미터 수신 · 타입 변환 · 계층 분리
패턴 ③ @RequestParam 타입변환★★★
// URL: /check?name=홍길동&age=17
@GetMapping("/check")
public String check(
    @RequestParam String name,
    @RequestParam int age,  // String→int 자동!
    Model model) {
  String r = (age >= 20) ? "성인" : "미성년";
  model.addAttribute("result", name + "은 " + r);
  return "result";
}
// [result.jsp]  ${result}
홍길동은 미성년
URL의 "17"(String) → int 17 자동 변환. age >= 20 조건 false → "미성년"
패턴 ④ VO 자동 매핑 (setter)★★
// form: <input name="name"><input name="age">
@GetMapping("/info")
public String info(MemberVO memberVo, Model model) {
    // form name값이 setName(), setAge()와
    // 매핑되어 자동으로 값이 들어옴!
    model.addAttribute("name", memberVo.getName());
    model.addAttribute("age", memberVo.getAge());
    return "info";
}
memberVo.getName() = "홍길동", getAge() = 30
form의 name="age" → setAge() 자동 호출. setter 이름과 input name이 반드시 일치해야 함
패턴 ⑤ @Autowired 계층 주입★★★
@Controller
public class MemberController {
    @Autowired
    MemberService memberService; // Spring 자동 주입
}
@Service
public class MemberService {
    @Autowired
    MemberDao memberDao;
}
@Service // DAO도 @Service로 Bean 등록
public class MemberDao {
    Map<String, MemberVO> memberDB = new HashMap<>();
}
@Autowired → new MemberService() 없이 Spring이 알아서 생성·주입
패턴 ⑥ Controller→Service→DAO 흐름★★★
// Controller
memberService.inputInfoConfirm(memberVo);

// MemberService
public int inputInfoConfirm(MemberVO memberVo) {
    System.out.println("[Service] inputInfoConfirm");
    System.out.println("name: " + memberVo.getName());
    memberDao.insertMember(memberVo);
    return 0;
}
// MemberDao
public void insertMember(MemberVO memberVo) {
    System.out.println("[DAO] inputInfo()");
    memberDB.put(memberVo.getName(), memberVo);
}
[Service] inputInfoConfirm
name: 홍길동
[DAO] inputInfo()
⚠
VO 자동 매핑: form의 <input name="age"> 에서 name 값과 VO의 setAge() 에서 "Age" 부분이 정확히 일치해야 한다. 오타 시 null 또는 0이 들어옴.

DB 연동 · JdbcTemplate · BCrypt · @PathVariable
MariaDB · update · query · queryForObject · PasswordEncoder
패턴 ⑦ JdbcTemplate INSERT★★★
String sql =
  "INSERT INTO member (id,pw,name,age,city)"
  + " VALUES (?,?,?,?,?)";

jdbcTemplate.update(sql,
    memberVo.getId(),
    memberVo.getPw(),
    memberVo.getName(),
    memberVo.getAge(),
    memberVo.getCity());
DB에 행 삽입. ? 자리에 순서대로 값 바인딩
update() = INSERT / UPDATE / DELETE 에 사용. 반환값은 영향받은 행 수(int)
패턴 ⑧ JdbcTemplate SELECT ALL★★★
String sql = "SELECT * FROM member";

jdbcTemplate.query(sql, (rs) -> {
    System.out.println(
        "id : " + rs.getString("id"));
    System.out.println(
        "age: " + rs.getInt("age"));
});
id : user1
age: 30
id : user2 ... (행 수만큼 반복)
람다 (rs) -> {} 안에서 ResultSet으로 각 행 처리. String → getString(), int → getInt()
패턴 ⑨ queryForObject — 로그인 조회★★★
String sql =
  "SELECT * FROM member WHERE id=? AND pw=?";
try {
    MemberVO m = jdbcTemplate.queryForObject(
        sql,
        (rs, rowNum) -> {
            MemberVO vo = new MemberVO();
            vo.setId(rs.getString("id"));
            vo.setName(rs.getString("name"));
            return vo;
        },
        memberVo.getId(), memberVo.getPw());
    return m;          // 로그인 성공
} catch (EmptyResultDataAccessException e) {
    return null;       // 로그인 실패
}
일치 행 있음 → MemberVO 반환 | 없음 → null 반환
결과 0건 → EmptyResultDataAccessException 발생! 반드시 try-catch로 null 처리
패턴 ⑩ BCryptPasswordEncoder★★
String raw = "1234";
String enc1 = passwordEncoder.encode(raw);
String enc2 = passwordEncoder.encode(raw);

System.out.println(enc1.equals(enc2));
// → false  (매번 다른 salt 적용)

System.out.println(
    passwordEncoder.matches("1234", enc1));
// → true

System.out.println(
    passwordEncoder.matches("5678", enc1));
// → false
false / true / false
같은 입력도 encode() 결과는 매번 다름. 비교는 반드시 matches(raw, encoded) 사용. equals()로 비교하면 항상 false!
패턴 ⑪ @PathVariable★★
// URL: /admin/books/3
@GetMapping("/admin/books/{bookId}")
public String detail(
    @PathVariable Long bookId,
    Model model) {
  System.out.println("책번호: " + bookId);
  model.addAttribute("bookId", bookId);
  return "bookDetail";
}
콘솔: 책번호: 3
JSP: ${bookId} → 3
URL 경로의 {bookId} 자리 값 "3" → Long 3 자동 변환

12주차
쿠키 · 세션 · 로그인 유지
HttpSession · Cookie · session.invalidate · redirect
패턴 ⑫ HttpSession 로그인★★★
@GetMapping("/member/login")
public String login(
    MemberVO memberVo, Model model,
    HttpSession session) {

  MemberVO m = memberService.loginConfirm(memberVo);

  if (m == null) {
    return "login_ng";
  } else {
    session.setAttribute("loginedMember", m);
    session.setMaxInactiveInterval(60 * 30);
    model.addAttribute("name", m.getName());
    return "login_ok";
  }
}
성공: session에 MemberVO 저장 → login_ok.jsp
실패: null 반환 → login_ng.jsp
패턴 ⑬ 로그아웃 + redirect★★★
@GetMapping("/member/logout")
public String logout(HttpSession session) {
  session.invalidate(); // 세션 전체 삭제
  return "redirect:/"; // 홈으로 리다이렉트
}

/* return "home"  과의 차이:
   "home"      → ViewResolver → home.jsp 렌더링
               → URL: /member/logout 유지
   "redirect:/" → 브라우저에 302 응답
               → 브라우저가 "/" 로 새 GET 요청
               → URL: "/" 로 변경됨           */
세션 삭제 후 브라우저 URL "/" 로 이동

비교 정리
자주 헷갈리는 개념 비교
↔ 나란히 보면 차이가 명확해지는 것들
↔ @RequestParam vs @PathVariable★★★
@RequestParam — 쿼리스트링
// URL: /check?name=홍길동&age=17 @GetMapping("/check") public String check( @RequestParam String name, @RequestParam int age) { ... }
? 뒤에 key=value 형태
@PathVariable — URL 경로값
// URL: /books/3 @GetMapping("/books/{bookId}") public String detail( @PathVariable Long bookId) { ... }
/경로/{값} 형태. {} 안 이름 = 파라미터명
↔ return "뷰명" vs return "redirect:URL"★★★
return "login_ok"
ViewResolver 거쳐 login_ok.jsp 렌더링 URL 변경 없음 (/member/login 유지) 서버 내부에서 처리
URL: /member/login (유지)
return "redirect:/"
브라우저에 HTTP 302 응답 브라우저가 "/" 로 새 GET 요청 발생 ViewResolver 거치지 않음
URL: / (변경됨)
로그아웃 후 redirect를 쓰는 이유: 새로고침 시 /logout이 재실행되는 것 방지
↔ Cookie vs Session★★★
구분CookieSession
저장 위치클라이언트 (브라우저)서버 (Tomcat 메모리)
보안취약 (브라우저에 노출)안전 (서버에 저장)
만료 설정setMaxAge(초) — 직접 지정setMaxInactiveInterval(초) 또는 브라우저 종료
저장 타입String만 가능Object 가능 (MemberVO 등)
삭제 방법setMaxAge(0)session.invalidate()
사용 예로그인 횟수 카운트로그인 상태 유지
↔ query() vs queryForObject()★★★
query() — 여러 행
// 결과 0~N행 처리 jdbcTemplate.query(sql, (rs) -> { // 각 행마다 실행됨 System.out.println(rs.getString("id")); }); // 결과 없어도 예외 없음
결과 없어도 예외 발생 안 함
queryForObject() — 정확히 1행
// 반드시 1행이어야 함 try { MemberVO m = jdbcTemplate .queryForObject(sql, (rs, rn) -> { ... }, id, pw); } catch (EmptyResultDataAccessException e) { return null; // 0건 시 예외! }
0건 → 예외 발생. try-catch 필수!

+ 버튼으로 정답 확인 | 먼저 스스로 풀어볼 것
★★★9주차 · ViewResolver
다음 코드에서 localhost:8090/ 접속 시 화면 출력값과 동작 원리를 단계별로 설명하라.
+
@Controller public class HomeController { @GetMapping("/") public String home(Model model) { model.addAttribute("msg", "Hello Spring!"); return "home"; } } // [home.jsp] <p>${msg}</p>
정답 및 원리
화면 출력: <p>Hello Spring!</p> ① "/" 요청 → DispatcherServlet 수신 ② HandlerMapping → @GetMapping("/") 을 가진 home() 탐색 ③ HandlerAdapter → home(Model) 실행 ④ model.addAttribute("msg", "Hello Spring!") → msg 키 저장 ⑤ return "home" → ViewResolver → prefix(/WEB-INF/views/) + "home" + suffix(.jsp) = /WEB-INF/views/home.jsp ⑥ home.jsp 실행 → ${msg} = "Hello Spring!" 치환 → 화면 출력
★★★10주차 · @RequestParam 타입변환
URL이 /member/checkAge?name=홍길동&age=17 일 때 화면 출력값과 원리를 설명하라.
+
@GetMapping("/member/checkAge") public String check(@RequestParam String name, @RequestParam int age, Model model) { String r = (age >= 20) ? "성인" : "미성년"; model.addAttribute("result", name + "은 " + r); return "result"; } // [result.jsp] <p>${result}</p>
정답 및 원리
화면 출력: <p>홍길동은 미성년</p> ① @RequestParam String name → "홍길동" ② @RequestParam int age → URL의 "17"(String)이 int 17로 자동 변환 ③ 17 >= 20 → false → r = "미성년" ④ model.addAttribute("result", "홍길동은 미성년") ⑤ return "result" → /WEB-INF/views/result.jsp ⑥ ${result} = "홍길동은 미성년" 출력 ★ age=20 이면? → r = "성인" → "홍길동은 성인"
★★★11주차 · queryForObject
DB에 id="user1", pw=BCrypt("1234") 가 저장되어 있다. 아래 코드에서 id="user1", pw="wrongpw" 로 호출 시 반환값과 이유를 설명하라.
+
String sql = "SELECT * FROM member WHERE id=? AND pw=?"; try { MemberVO m = jdbcTemplate.queryForObject(sql, (rs, rowNum) -> { MemberVO vo = new MemberVO(); vo.setId(rs.getString("id")); vo.setName(rs.getString("name")); return vo; }, "user1", "wrongpw"); return m; } catch (EmptyResultDataAccessException e) { return null; }

★★★11주차 · BCrypt encode/matches
아래 코드의 (A), (B), (C) 출력값을 각각 쓰고 이유를 설명하라.
+
String enc1 = passwordEncoder.encode("1234"); String enc2 = passwordEncoder.encode("1234"); System.out.println(enc1.equals(enc2)); // (A) System.out.println(passwordEncoder.matches("1234", enc1)); // (B) System.out.println(passwordEncoder.matches("5678", enc1)); // (C)
정답 및 원리
(A) false BCrypt는 encode() 시 매번 다른 랜덤 salt 생성 → enc1 ≠ enc2 (같은 "1234"여도) (B) true matches("1234", enc1): enc1 안에 내장된 salt를 꺼내 "1234"에 적용하여 해시 비교 → 원문이 같으므로 true (C) false "5678"의 해시가 enc1과 다름 → false 핵심: equals()로 두 encoded 값 비교 → 항상 false 비교는 반드시 matches(rawPw, encodedPw) 사용!

★★10주차 · VO 자동 매핑 함정
form에서 <input name="userName"> 으로 전송했는데 MemberVO의 name 필드에 값이 들어오지 않는다. 이유와 해결 방법을 설명하라.
+
<!-- form --> <input type="text" name="userName"> <!-- userName --> // MemberVO public class MemberVO { String name; // 필드명: name public void setName(String name) { this.name = name; } // setter: setName → "name" 에 매핑됨 }
정답 및 원리
이유: VO 자동 매핑은 form의 input name 값을 setXxx() 형태의 setter와 매핑함. input name="userName" → setUserName() 을 찾음 MemberVO에는 setName()만 있고 setUserName()이 없음 → 매핑 실패 → name 필드에 null이 들어옴 해결 방법 (둘 중 하나): ① form을 수정: name="userName" → name="name" ② VO에 setter 추가: public void setUserName(String name) { this.name = name; } 핵심: form의 input name 값 = set + 첫글자대문자(name값) name="age" → setAge() 와 매핑 name="userName" → setUserName() 와 매핑
