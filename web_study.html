10주차 2차시 · Service & DAO 구현
@Service 빈 등록 & @Autowired 주입
// MemberService.java
@Service   // 스프링 컨테이너가 자동 생성
public class MemberService {

    @Autowired
    MemberDao memberDao;   // DAO 자동 주입

    public int signUpConfirm(MemberVO memberVo) {
        System.out.println("[MemberService] signUpConfirm");
        memberDao.insertMember(memberVo);
        return 0;
    }
}
// MemberController.java
@Controller
public class MemberController {

    @Autowired
    MemberService memberService;   // Service 자동 주입

    @PostMapping("/signUpConfirm")
    public String signUpConfirm(MemberVO memberVo, Model model) {
        memberService.signUpConfirm(memberVo);
        return "sign_up_ok";
    }
}
콘솔 출력: [MemberService] signUpConfirm [MemberDao] insertMember
@Autowired 동작: 스프링 컨테이너가 해당 타입의 빈을 찾아 자동으로 필드에 주입. @Service, @Repository 붙은 클래스만 빈으로 등록됨
@Repository DAO 구현 (DB 없이 HashMap 사용)
@Repository
public class MemberDao {

    HashMap<String, MemberVO> memberDB = new HashMap<>();

    public void insertMember(MemberVO memberVo) {
        System.out.println("[MemberDao] insertMember");
        memberDB.put(memberVo.getId(), memberVo);
    }

    public MemberVO selectMember(MemberVO memberVo) {
        MemberVO loginedMember = memberDB.get(memberVo.getId());
        if (loginedMember != null
            && memberVo.getPw().equals(loginedMember.getPw())) {
            return loginedMember;
        }
        return null;
    }
}

11주차 1차시 · 데이터베이스 연동 (JdbcTemplate)
JdbcTemplate - INSERT (update 메소드)
@Repository
public class MemberDao {

    @Autowired
    JdbcTemplate jdbcTemplate;   // 스프링이 자동 주입

    public void insertMember(MemberVO memberVo) {
        String sql = "INSERT INTO member (id, pw, name, age, city) VALUES (?, ?, ?, ?, ?)";
        jdbcTemplate.update(sql,
            memberVo.getId(), memberVo.getPw(),
            memberVo.getName(), memberVo.getAge(), memberVo.getCity());
    }
}
jdbcTemplate.update() → INSERT, UPDATE, DELETE에 사용. ?는 순서대로 파라미터와 매핑
JdbcTemplate - SELECT (query 메소드)
public void printAllMember() {
    String sql = "SELECT * FROM member";

    jdbcTemplate.query(sql, (rs, rowNum) -> {
        MemberVO vo = new MemberVO();
        vo.setId(rs.getString("id"));
        vo.setPw(rs.getString("pw"));
        vo.setName(rs.getString("name"));
        vo.setAge(rs.getInt("age"));
        vo.setCity(rs.getString("city"));
        return vo;
    });
}
jdbcTemplate.query() → SELECT 여러 행. 람다 (rs, rowNum) → { } 로 각 행을 VO로 변환
JdbcTemplate - SELECT WHERE (조건 조회 + 비밀번호 암호화)
public MemberVO selectMember(MemberVO memberVo) {
    String sql = "SELECT * FROM member WHERE id = ?";

    List<MemberVO> members = jdbcTemplate.query(sql, (rs, rowNum) -> {
        MemberVO vo = new MemberVO();
        vo.setId(rs.getString("id"));
        vo.setPw(rs.getString("pw"));
        // ... 나머지 필드
        return vo;
    }, memberVo.getId());   // ← ? 에 들어갈 값

    if (members.isEmpty()) return null;

    // 비밀번호 암호화 비교
    if (passwordEncoder.matches(memberVo.getPw(), members.get(0).getPw())) {
        return members.get(0);
    }
    return null;
}
비밀번호는 DB에 암호화 저장 → 비교 시 passwordEncoder.matches(평문, 암호화값) 사용. equals() 쓰면 항상 불일치!
JdbcTemplate 메소드 비교
메소드용도반환값
update(sql, ...)INSERT / UPDATE / DELETEint (영향받은 행 수)
query(sql, (rs,n)->{})SELECT 다중 행List<T>
queryForObject(sql, type)SELECT 단일 값T (없으면 예외)
12주차 1차시 · 쿠키와 세션
왜 필요한가 — HTTP의 비연결성(Stateless) 문제
① 로그인 요청 → 서버: 안녕하세요 → ② 다음 요청 → 서버: 누구세요? 😱
HTTP의 특성: 비연결(Connectionless) + 무상태(Stateless) → 매 요청마다 새로운 클라이언트로 인식 → 쿠키/세션으로 해결
세션 — 로그인 상태 유지
@GetMapping("/member/login")
public String memberLogin(MemberVO memberVo, Model model,
                          HttpSession session) {  // 세션 파라미터 추가
    MemberVO loginedMember = memberService.loginConfirm(memberVo);

    if (loginedMember == null) {
        return "login_ng";
    } else {
        // 세션에 로그인 정보 저장
        session.setAttribute("loginedMember", loginedMember);
        session.setMaxInactiveInterval(60 * 30);  // 30분 유지
        model.addAttribute("name", loginedMember.getName());
        return "login_ok";
    }
}
세션 — 로그아웃
@GetMapping("/member/logout")
public String memberLogout(HttpSession session) {
    session.invalidate();         // 세션 전체 제거
    return "redirect:/";           // 홈으로 리다이렉트
}
session.invalidate() → 세션에 저장된 모든 정보 삭제 (로그아웃 처리)
return "/"와 return "redirect:/"는 다름! "/"는 /WEB-INF/views/.jsp를 찾고, "redirect:/"는 URL "/" 로 실제 리다이렉트
쿠키 — 방문 횟수 카운트 구현
// 쿠키 읽기
public int getLoginCount(HttpServletRequest request) {
    Cookie[] cookies = request.getCookies();
    if (cookies == null) return 0;

    for (Cookie cookie : cookies) {
        if (COOKIE_LOGIN_COUNT.equals(cookie.getName())) {
            return Integer.parseInt(cookie.getValue());
        }
    }
    return 0;
}

// 쿠키 쓰기
public int incrementLoginCount(HttpServletRequest request,
                               HttpServletResponse response) {
    int newCount = getLoginCount(request) + 1;

    Cookie cookie = new Cookie(COOKIE_LOGIN_COUNT,
                                String.valueOf(newCount));
    cookie.setPath("/");
    cookie.setMaxAge(COOKIE_MAX_AGE_SECONDS);  // 유효시간 설정
    response.addCookie(cookie);               // 응답에 쿠키 추가

    return newCount;
}
쿠키 핵심: 읽기는 request.getCookies(), 쓰기는 response.addCookie(). 서버→브라우저→서버 순환
쿠키 vs 세션 핵심 비교

🍪 쿠키 (Cookie)

  • 저장 위치: 클라이언트(브라우저)
  • 보안: 낮음 (브라우저에 노출)
  • 유지: 설정한 MaxAge 동안
  • 브라우저 기준으로 동작
  • 서버 부하 없음
  • 용도: 방문 횟수, 자동완성, 장바구니

🔐 세션 (Session)

  • 저장 위치: 서버
  • 보안: 높음 (서버에만 저장)
  • 유지: 브라우저 닫거나 invalidate()
  • 사용자 계정 기준으로 동작
  • 서버 메모리 사용
  • 용도: 로그인 상태 유지, 민감 정보
쿠키는 "브라우저 기준"! 같은 브라우저에서 다른 아이디로 로그인해도 같은 쿠키 사용 → loginCount가 아이디 상관없이 증가

⚔ 헷갈리는 개념 완전 비교
@RequestParam vs @PathVariable

@RequestParam

  • URL: /search?name=홍길동
  • 쿼리스트링 또는 폼 데이터
  • HTML <form> 제출 시 사용
  • required=false 가능

@PathVariable

  • URL: /member/42
  • URL 경로의 일부
  • RESTful API 스타일
  • @GetMapping("/m/{id}") 필요
// @RequestParam
@GetMapping("/search")
public String search(@RequestParam String name) {...}
// 접근: /search?name=홍길동

// @PathVariable
@GetMapping("/member/{id}")
public String detail(@PathVariable int id) {...}
// 접근: /member/42
return "viewName" vs return "redirect:/url"

return "viewName"

  • ViewResolver를 통해 JSP 렌더링
  • "home" → /WEB-INF/views/home.jsp
  • URL 주소 변화 없음
  • Model 데이터 유지됨

return "redirect:/url"

  • 새 URL로 재요청 (HTTP 302)
  • "redirect:/" → "/" 재요청
  • URL 주소 변화 있음
  • Model 데이터 초기화됨
@Controller vs @Service vs @Repository
애너테이션레이어주요 역할특이사항
@Controller웹 레이어HTTP 요청/응답 처리View 이름 반환, @RequestParam 사용
@Service서비스 레이어비즈니스 로직@Controller에서 @Autowired로 주입
@Repository데이터 레이어DB CRUD@Service에서 @Autowired로 주입
셋 다 스프링 컨테이너에 빈(Bean)으로 등록됨. @Autowired는 같은 타입의 빈을 찾아 자동 주입
jdbcTemplate.update() vs jdbcTemplate.query()
메소드사용 상황반환 타입예시 SQL
update(sql, ...)데이터 변경int (영향 행 수)INSERT, UPDATE, DELETE
query(sql, mapper, ...)여러 행 조회List<T>SELECT * FROM member
queryForObject()단일 값 조회T (없으면 예외)SELECT COUNT(*)
session.setAttribute() vs model.addAttribute()

session.setAttribute()

  • 서버 세션에 저장
  • 여러 요청에 걸쳐 유지
  • 로그인 상태, 장바구니
  • invalidate()로 삭제

model.addAttribute()

  • 현재 요청의 Model에 저장
  • 이번 요청-응답 1회만 유효

★★★★★ MVC 흐름 return "home" 이 왜 home.jsp를 여는가 ▼
ViewResolver 공식
prefix  +  return값  +  suffix
/WEB-INF/views/  +  "home"  +  .jsp
=  /WEB-INF/views/home.jsp
// application.properties
spring.mvc.view.prefix=/WEB-INF/views/
spring.mvc.view.suffix=.jsp

// HomeController.java
@GetMapping("/")
public String home() {
    return "home";
}
동작 원리
① 브라우저 → / 요청
② DispatcherServlet 수신
③ HandlerMapping:
  @GetMapping("/") 찾음
④ home() 실행
⑤ return "home"
⑥ ViewResolver:
  prefix + "home" + suffix
⑦ → home.jsp 렌더링
1
prefix/suffix는 application.properties에서 설정한다. 이 값이 없으면 return 값을 그대로 URL로 해석하려 해서 오류 발생.
2
ViewResolver가 문자열을 조립해 실제 파일 경로를 만든다. 개발자가 직접 경로를 쓰지 않아도 되는 이유.
3
return "redirect:/"는 ViewResolver를 거치지 않고 브라우저에게 "/"로 다시 요청하라는 HTTP 302 응답을 보낸다.
return "redirect:/home" 이면 home.jsp가 아니라 "/" 로 리다이렉트 후 다시 @GetMapping("/home") 을 찾는다. ViewResolver를 거치지 않음!

// 요청: /member/login?id=admin&pw=1234

// ① Controller
@GetMapping("/member/login")
public String memberLogin(MemberVO vo) {
  System.out.println("[Controller]");
  memberService.loginConfirm(vo); // ②
  return "login_ok";
}
// ② Service
@Service
public class MemberService {
  public MemberVO loginConfirm(MemberVO vo) {
    System.out.println("[Service]");
    return memberDao.selectMember(vo); // ③
  }
}
// ③ DAO
@Repository
public class MemberDao {
  public MemberVO selectMember(MemberVO vo) {
    System.out.println("[Dao]");
    return null;
  }
}
콘솔 출력 (순서 보장)
[Controller]
[Service]
[Dao]

반환값 흐름 (역방향):
Dao → null 반환
Service → null 반환
Controller → "login_ok"
1
자바 메소드 호출은 동기(순차) 실행. memberService.loginConfirm() 호출 시 그 안이 완전히 끝날 때까지 Controller는 대기.
2
Service 내부에서 memberDao.selectMember() 호출 시 마찬가지로 DAO가 끝날 때까지 대기. → 호출 순서 = 출력 순서.
3
@Autowired가 각 레이어 간 의존 관계를 연결. Controller에 @Autowired MemberService, Service에 @Autowired MemberDao 없으면 NullPointerException.
4
@Autowired가 동작하려면 해당 클래스가 반드시 @Service 또는 @Repository로 스프링 빈 등록되어 있어야 한다.
@Service를 빠트리면 @Autowired가 주입할 빈을 못 찾아 NoSuchBeanDefinitionException 발생. 출력이 아예 안 됨.
출력 순서 = 메소드 호출 순서 = Controller → Service → DAO. 반환값 전달은 반대 방향(DAO → Service → Controller).
★★★★★ DB 연동 jdbcTemplate.update() vs .query() — 왜 다른 메소드를 쓰는가 ▼ 
// INSERT — jdbcTemplate.update()
String sql1 = "INSERT INTO member"
           + " (id,pw,name) VALUES (?,?,?)";
jdbcTemplate.update(sql1,
    vo.getId(), vo.getPw(), vo.getName());

// SELECT — jdbcTemplate.query()
String sql2 = "SELECT * FROM member";
List<MemberVO> list =
    jdbcTemplate.query(sql2, (rs, rowNum) -> {
        MemberVO v = new MemberVO();
        v.setId(rs.getString("id"));
        v.setName(rs.getString("name"));
        return v;
    });
각 메소드가 반환하는 것
update():
int → 영향받은 행 수
(1이면 성공, 0이면 실패)

query():
List<MemberVO>
→ 각 행을 VO로 변환한 목록

rs = 현재 행 데이터
rowNum = 현재 행 번호
1
update()는 데이터를 변경(INSERT/UPDATE/DELETE)할 때 사용. DB에 변경이 발생하는 모든 SQL에 대응.
2
query()는 데이터를 조회(SELECT)할 때 사용. 람다 (rs, rowNum) → { }가 각 행마다 1번씩 실행되어 VO를 만들고 List에 추가.
3
rs.getString("컬럼명")은 문자열, rs.getInt("컬럼명")은 정수 반환. DB 컬럼 타입과 일치시켜야 함.
4
WHERE 조건의 ?는 메소드 마지막 인자로 순서대로 전달. query(sql, mapper, 파라미터1, 파라미터2)
INSERT에 query()를 쓰면 행 수 반환이 아닌 List를 반환하려 하므로 타입 오류. 반드시 update() 사용.
★★★★★ 세션 session.setAttribute() 가 로그인 상태를 유지하는 원리 ▼
세션 동작 공식
최초 요청 → 서버가 세션ID 발급 → 브라우저 쿠키에 JSESSIONID 저장

// 로그인 성공 시
@GetMapping("/member/login")
public String login(MemberVO vo,
    Model model, HttpSession session) {

  MemberVO member =
    memberService.loginConfirm(vo);

  if (member == null) {
    return "login_ng";
  }
  session.setAttribute(
    "loginedMember", member);
  session.setMaxInactiveInterval(
    60 * 30);
  model.addAttribute("name",
    member.getName());
  return "login_ok";
}
로그인 성공 흐름
loginConfirm() → MemberVO 반환

session["loginedMember"]
= MemberVO{id="hong"...}

30분 비활성 시 자동 만료

model["name"] = "홍길동"

→ login_ok.jsp 렌더링
→ ${name}님 로그인 성공!

로그아웃 시:
session.invalidate() 호출
→ 세션 전체 삭제
1
HttpSession은 서버 메모리에 저장. 브라우저는 JSESSIONID 쿠키만 가지고 있고 실제 데이터는 서버에 있음.
2
session.getAttribute("loginedMember")로 다른 페이지에서도 로그인 정보 조회 가능. 로그인 체크에 활용.
3
session.invalidate()는 세션에 저장된 모든 데이터를 삭제. 로그아웃 처리의 핵심.
4
setMaxInactiveInterval(초)는 마지막 요청으로부터 해당 시간 동안 무활동 시 세션 자동 만료. 60*30 = 1800초 = 30분.
model.addAttribute()는 현재 요청·응답 1번만 유효. 다음 요청에서 사라짐. 로그인 상태처럼 여러 요청에 걸쳐 유지할 데이터는 반드시 session 사용.

}

★★★ Docker Docker Compose에서 localhost 대신 db를 쓰는 이유 ▼ 
# docker-compose.yml
services:
  app:
    ...
    environment:
      # ❌ 잘못된 설정
      SPRING_DATASOURCE_URL:
        jdbc:mariadb://localhost:3306/lib
      # ✅ 올바른 설정
      SPRING_DATASOURCE_URL:
        jdbc:mariadb://db:3306/lib
  db:
    image: mariadb:latest
    ...
왜 다른가
localhost 사용 시:
app 컨테이너 내부의
자기 자신 127.0.0.1을 가리킴
→ MariaDB 없음
→ Connection refused 오류

db 사용 시:
Docker 내부 DNS가
"db" → db 컨테이너 IP 매핑
→ 정상 연결
1
Docker Compose는 같은 compose 파일의 서비스들을 동일한 가상 네트워크에 연결. 서비스명이 곧 호스트명.
2
각 컨테이너 내부에서 localhost는 자기 자신을 가리킴. 다른 컨테이너에 접근하려면 서비스명 사용.
3
depends_on: db: condition: service_healthy는 DB가 완전히 준비된 후 앱을 시작. 없으면 DB 초기화 전에 앱이 연결 시도 → 오류.
서비스명(db)은 docker-compose.yml에서 정의한 이름과 정확히 일치해야 함. 대소문자 구분 있음.
